# AI 智能选品系统 — 四期立项方案

## Context

公司主营跨境电商（工业科学/家居/汽配），重资产海外仓囤货模式，SKU 3万+。当前选品依赖人工在卖家精灵 SaaS 上手动筛选，效率低且决策质量不稳定。本项目目标：构建 AI 辅助选品系统，逐步从「人工筛选」走向「机器推荐+人工复核」，降低选品失误率、提升开品效率。

立项时间：2026-06-15 | 一期交付：2026-09-30

### 技术约束

| 项目 | 现状 |
|------|------|
| 交付形式 | 先独立站点，后迁入 ERP |
| 技术栈 | Java（Spring Boot）+ 前端框架（Vue3/React） |
| 选品系统团队 | 2 名 Java 后端 + 1 名前端，7.1 全职介入 |
| ERP 团队 | 2 名 Java 后端，负责 ERP 产品库日常研发，8 月可介入接口对接 |
| ERP 对接 | 需 ERP 团队开发「创建开品单」接口 |
| 卖家精灵 API | 商务洽谈中，选产品 API + Proboost 类目快照分开购买 |
| 内部数据 | 侵权词库 2万条（已有）、产品库图片（已有） |

## 一期：选品闭环 MVP（6.15 — 9.30）

### 时间分配

| 阶段 | 时间 | 人员 | 产出 |
|------|------|------|------|
| 产品设计 + 技术方案 | 6.15 - 6.30（2周） | 产品经理（你）| PRD、技术方案、API 文档整理、原型确认 |
| 开发阶段 | 7.1 - 9.15（11周） | 2 Java + 1 前端 | 系统开发 |
| ERP 联调 | 8.1 - 9.15（7周，与开发并行） | ERP 团队 2 人 | 开品接口开发 + 联调 |
| 测试 + 试运行 | 9.15 - 9.30（2周） | 全员 | 内部验收、选品组试用 |

### 目标

跑通「筛选 → 风控 → 结果池 → 查重去废 → 推送 ERP 开品」完整闭环，替代人工手动操作卖家精灵 SaaS。

### 功能模块

| 模块 | 说明 | 核心能力 |
|------|------|----------|
| 选产品 | 对接卖家精灵「选产品」API | 64 参数筛选、模式保存、站点切换 |
| 内部风控过滤 | 自动前置过滤 | 侵权词库匹配 + 以图搜图去重（与 ERP 已有 SKU 比对） |
| 选品结果池 | 汇集筛选结果 | 列表管理、标记状态、人工复核打分 |
| 查重&作废 | 结果池内二次去重 | 重复 ASIN 合并、人工标记作废、批量操作 |
| 推送开品 | 打通 ERP | 选定商品 → 推送 ERP 产品库 → 自动填充可映射字段 → 开发专员补全资料 → 提交开品 |

### 里程碑

| M2 核心链路可跑 | 8.31 | 选产品→风控过滤→结果池→查重作废 全链路跑通 |
| M3 ERP 打通 | 9.15 | 结果池→推送 ERP 创建开品单接口联调通过 |
| M4 一期上线 | 9.30 | 内部试运行，选品组实际使用 |

## 二期：竞品监控与洞察（10.15 — 11.30，7 周）

### 目标

购买 Proboost 类目快照 API，实现竞品监控→异动预警→快速响应策略闭环。

### 功能模块

| 模块 | 说明 |
|------|------|
| 类目快照订阅 | 设置监控的子类目 + 竞品 ASIN，定时拉取快照数据 |
| 子类目对比 | 类目维度横向对比（BSR 分布、价格带、新品占比、卖家集中度） |
| 竞品异动监控 | 价格变动、排名跳变、新增变体、断货、差评激增等事件检测 |
| 洞察响应策略 | 异动结果关联行动：<br>- 以图搜图查内部全球库存 SKU → 运营快速行动<br>- 加入选品结果池 → 走开品流程<br>- 评论情绪分析 → 差异化机会<br>- 关键词趋势分析 → Listing 优化建议 |

### 里程碑

| 节点 | 日期 | 交付物 |
|------|------|--------|
| M5 Proboost 对接 | 10.31 | 类目快照 API 联调通过，数据入库 |
| M6 监控上线 | 11.15 | 子类目对比 + 竞品异动检测可用 |
| M7 二期交付 | 11.30 | 洞察→行动策略全链路跑通 |

## 三期：AI 自动选品算法（12.15 — 次年 1.30，7 周）

### 目标

基于一期人工选品经验沉淀 + 内外部数据融合，初步实现 AI 自动跑分推荐。

### 核心能力

| 能力 | 说明 |
|------|------|
| 数据融合 | 卖家精灵数据包 + Proboost 数据包 + 内部产品动销 + 库存压力 + 供应商支持情况 |
| 权重模型 | 多维度加权评分：市场容量、竞争度、利润率、供应链可行性、风险系数 |
| 自动选品 | 定时/按需触发 → AI 自动对比内外部数据 → 输出评分排序 → 推入选品结果池 |
| 经验回溯 | 一期人工选品结果（成功/失败）作为模型训练/校准的标注数据 |

### 里程碑

| 节点 | 日期 | 交付物 |
|------|------|--------|
| M8 数据仓库 | 12.31 | 内外部数据入库、清洗、可查询 |
| M9 算法 V1 | 1.15 | 评分模型上线，能产出推荐列表 |
| M10 三期交付 | 1.30 | AI 自动选品→结果池链路跑通，人工校准机制就位 |

## 四期：全自动化 + 货源寻源（2 月+，持续迭代）

### 目标

完善选品+洞察+AI 算法，对接 1688 实现货源寻源自动化。

### 核心能力

| 能力 | 说明 |
|------|------|
| 1688 API 对接 | 关键词搜品、供应商筛选、价格区间过滤 |
| 自动询价 | 选品结果池确认后 → 自动向匹配供应商发起询价 |
| 供应商评估 | 历史合作评分、响应速度、品质稳定性加权 |
| 选品→寻源→开品 全自动 | 减少人工介入节点，仅在关键决策点保留审批 |

## 立项前必须锁定的事项（6.15 — 6.30 产品准备期）

1. **卖家精灵合同** — 6月底前签约拿到测试 key（最高优先级）
2. **PRD 定稿** — 选产品、结果池、风控过滤、推送开品四个模块的详细需求
3. **ERP 接口规范** — 提前与 ERP 团队对齐「创建开品单」的字段映射和接口协议
4. **以图搜图对接** — 确认 ERP 现有以图搜图服务的调用方式，选品系统直接复用
5. **开发环境 & 基础设施** — 代码仓库、CI/CD、测试环境、数据库选型（MySQL + Redis + 向量库）
